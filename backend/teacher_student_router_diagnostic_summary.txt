Concise diagnostic summary

High-risk endpoints under repeated traffic
1) backend/routers/teacher.py -> GET /api/teacher/distribution
   Function: teacher_distribution()
   Why risky:
   - Performs a Firestore query per assigned batch for paid payments:
     db.collection("payments").where(...).where(...).where(...).where("batch_id","==", bid).stream()
   - Then performs repeated user document lookups inside nested loops:
     db.collection("users").document(tid).get() is called once per teacher per batch, and again once per teacher per date-group.
   - Builds all_payments in memory, then repeatedly re-filters it in Python by batch/date.
   - No pagination or limits anywhere; response size can grow with all paid payments for the month.
   Production impact:
   - Heavy synchronous Firestore I/O inside an async route can block the event loop.
   - Latency and memory use increase with number of batches, payments, and dates.
   - On Render, enough slow requests can make health checks/timeouts more likely, which can lead to worker restarts or graceful shutdowns.

2) backend/routers/student.py -> GET /api/student/leaderboard
   Function: student_get_leaderboard()
   Why risky:
   - Loads every payment for a batch/month/year into memory with .stream(), then filters/sorts in Python.
   - Separately counts students by streaming all matching users and summing them in Python.
   - For top 5, does per-student document fetches:
     db.collection("users").document(p["student_id"]).get()
   - Then issues another unbounded stream of all payments for the current student just to compute available_months.
   - Calls sp.to_dict() multiple times inside a set comprehension for the same document.
   Production impact:
   - Multiple Firestore round trips and unbounded reads per request.
   - Slow endpoint for larger batches/payment histories.
   - Because the handler is async but uses blocking SDK calls, concurrent requests can stall other requests.

3) backend/routers/teacher.py -> GET /api/teacher/payments
   Function: teacher_get_payments()
   Why risky:
   - Query streams all payments for a batch, optionally month/year filtered, with no pagination or limit.
   - Loads full result set into memory, then sorts in Python.
   - Enriches student names via db.get_all() over all distinct student IDs; chunked by 100, but still unbounded overall.
   Production impact:
   - If a batch has a long payment history and month/year are omitted, one request can read the entire batch payment history.
   - Large response bodies and blocking Firestore calls can increase request duration and tie up workers.

Medium-risk endpoints
4) backend/routers/student.py -> GET /api/student/payments
   Function: student_get_payments()
   Why risky:
   - Streams all payments for the logged-in student with no pagination/limit.
   - Does one user document get per distinct requested_by_teacher to resolve teacher names.
   - Sorts in Python after loading all rows.
   Production impact:
   - Usually smaller than teacher endpoints, but still unbounded per student account.
   - Repeated traffic from active users can create unnecessary blocking I/O and response growth over time.

5) backend/routers/teacher.py -> GET /api/teacher/batches
   Function: teacher_get_batches()
   Why risky:
   - After streaming assigned batches, performs an additional streamed users query per batch to count students.
   Production impact:
   - N+1 query pattern; becomes slow for teachers assigned to many batches.
   - Less likely to crash alone, but contributes to cumulative latency under load.

Mutation endpoints with blocking side effects
6) backend/routers/student.py -> POST /api/student/payments/{payment_id}/upload
   Function: student_upload_screenshot()
   Why risky:
   - Reads entire uploaded file into memory: contents = await file.read()
   - Calls cloudinary.uploader.upload(...) synchronously inside async handler.
   - Then performs Firestore update and notification sends inline.
   - No outer try/except around Firestore update or notification calls after upload.
   Production impact:
   - Large uploads can spike memory.
   - Blocking Cloudinary/network work can stall the event loop.
   - If notification code raises after upload succeeds, request can fail after partial side effects, causing retries/duplicate work.

7) backend/routers/teacher.py -> POST /api/teacher/offline-request
   Function: teacher_offline_request()
   Why risky:
   - Synchronous Firestore reads/writes plus synchronous notification calls in async handler.
   - No try/except around Firestore update/add or notification calls.
   Production impact:
   - Exceptions bubble out as 500s and may trigger client retries.
   - Under repeated use, blocking notification/network work can slow unrelated requests.

Cross-cutting application-level concern seen from these routers
8) Firestore Admin SDK usage is synchronous everywhere
   Files:
   - backend/routers/student.py
   - backend/routers/teacher.py
   - also supported by backend/database.py import-time db client initialization
   Why risky:
   - All route handlers are declared async, but Firestore Python Admin client calls (.get(), .stream(), .update(), .add(), db.get_all()) are blocking.
   Production impact:
   - Under concurrent traffic, event loop threads spend time blocked on network I/O.
   - This can manifest as slow health responses, request queueing, and worker churn on Render.

9) Background scheduler adds more synchronous Firestore/network load
   File: backend/scheduler.py
   Why relevant to these endpoints:
   - run_scheduler() is started at app startup from backend/main.py using asyncio.create_task(run_scheduler()).
   - send_daily_due_reminders() streams all unpaid payments and calls notify_user for every student with dues.
   - Current file content also prints “TEST MODE ENABLED” though sleep still targets next 10:00 IST; regardless, it is an always-running background task in the same process.
   Production impact:
   - Shares the same process/resources as API routes.
   - If notify_user or Firestore calls block, API responsiveness worsens during reminder runs.

Most likely matches to the reported traffic pattern
- /api/teacher/payments: likely culprit if frontend polls or loads this without month/year, because it can stream an entire batch payment history and enrich it synchronously.
- /api/student/payments: less severe but still unbounded and blocking.
- /api/teacher/distribution and /api/student/leaderboard: strongest application-level hotspots due to repeated full scans, nested document fetches, and Python-side aggregation.

Bottom-line assessment
- I do not see route code that directly requests app shutdown.
- I do see several expensive, unbounded, synchronous Firestore/network code paths inside async endpoints, plus inline Cloudinary/notification work.
- Under repeated API traffic, these endpoints can become slow enough to cause health-check failures, timeouts, or resource pressure, which is consistent with a platform like Render gracefully stopping and restarting the process.
